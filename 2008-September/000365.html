Hi Khaldon

Thanks for you feedback, we appreciate it. To answer your question the
RTSP attached to the 200 OK follows RFC 2017 by inserting the RTSP address
as a message/external-body in the Content-Type of the reply.

When the client receives this it extracts the RTSP url and uses libvlc to
establish the RTSP session with the video server and begin viewing the
stream.

The solution is not fully conformant with ETSI TISPAN standards.The invite
should in fact be forwarded onto the Media Control Function by the
Application Server once it has checked that the user is allowed to access
the content. Also the use of the message/external-body is not correct and
the SDP should instead contain and "m=" line conveying the RTSP details in
either the clients initial invite or in the 200 OK sent by the MCF in
response to the invite.

We are currently looking at implementing an MCF and extending the client
to to be ETSI TS compliant.
