Sorry to be drumming on about this, but have you checked in the client
preferences to enable video calling ?

I'll take a look as soon as I can, but please check that video calling is
enabled.

Options -> Preferences -> Profile -> Video Calling -> Enabled.

If this option is not enabled, the client doesn't add video information to
the SDP part of the invite. Can you check this for me, and perhaps email a
full list of the output messages of the client and server ?

> > Hi Shan
> >
> > To be honest, the IPTV server wasn't designed to only stream audio -
> > Hence we reject any calls that do not have video enabled. The rest of the
> > messages provided by the server (e.g. ffenc_h263p etc) is what you should
> > see when using a compatible file (with more information instead of NULL if
> > there is actually a video component).
> >
> > I think the difficulties you are experiencing have to do with the
> > gstreamer plugins you are running.
> >
> > What version are you using ?
> >
> > Does a command line like the following work ?
> >
> > gst-launch-0.10 filesrc location=/path/to/file.avi ! decodebin name=dbin
> > ! audioconvert ! audioresample ! queue ! ffenc_mp2 !  rtpmpapay ! udpsink
> > port=2222 dbin. ! ffmpegcolorspace ! queue ! ffenc_h263p ! rtph263ppay !
> > udpsink port=2223
> >
> > This command starts streaming the file - audio to port 2222 and video to
> > port 2223.
> >
> > To receive the video you can use
> >
> >  gst-launch-0.10 udpsrc port=2223 caps="application/x-rtp,
> > media=(string)video, clock-rate=(int)90000, encoding-name=(string)H263-1998,
> > payload=(int)96, ssrc=(guint)647401835, clock-base=(guint)3452060272,
> > seqnum-base=(guint)60746"  ! rtph263pdepay ! ffdec_h263 ! ffmpegcolorspace !
> > xvimagesink sync=false
> >
> > Can you try these out and get back to me ?
> >
> > (What happens if you run the client and enable video calling - I havn't
> > done this myself so I dont know if it will work or not. It can be found in
> > the preferences.)
